# Acme Cloud - Grounded Reply Assistant

## Overview
This project implements an intelligent customer support assistant for a fictional company, Acme Cloud. The system is designed to provide accurate, policy-compliant answers to user queries, execute mock transactions, and gracefully refuse off-topic questions. The assistant relies on Retrieval-Augmented Generation (RAG) and function calling to ground its responses strictly in the provided company handbook.

## Architecture and Key Components

### 1. Retrieval-Augmented Generation (RAG) Pipeline
The system uses a local embedding model (`nomic-ai/nomic-embed-text-v1.5` via `SentenceTransformers`) to convert user queries and handbook text into semantic vectors.
- **Vector Database:** `ChromaDB` is used in-memory to store and query the embeddings using cosine similarity.
- **Granular Chunking Strategy:** The handbook is dynamically chunked by section headers (`##`). For highly specific policies (like bulleted lists), the chunking logic splits individual bullet points while retaining their parent headers. This increases semantic density and ensures high precision during retrieval.
- **Similarity Threshold:** A hard threshold of 0.45 is enforced. If no chunk meets this threshold, the system automatically refuses to answer, effectively preventing hallucinations on off-topic questions.

### 2. Model Fallback Mechanism
To ensure high availability and handle rate limits, the system implements a fallback architecture:
- **Primary Model:** `openai/gpt-oss-20b:groq` (via HuggingFace). This larger model excels at isolating multiple intents in complex queries and accurately populating structured outputs.
- **Fallback Model:** `llama-3.1-8b-instant` (via Groq). If the primary model fails or encounters a credit deficit, the system gracefully falls back to this secondary model to maintain uptime.

### 3. Structured Outputs
The system uses the `Instructor` library built on top of `Pydantic`. Instead of parsing raw text, the model is forced to return a strictly typed JSON object (`GroundedReply`). This object includes:
- A boolean flag indicating if the query was answered.
- A confidence score (0.0 to 1.0) generated by the model.
- The final response text.

### 4. Function and Tool Calling
The assistant acts as an autonomous agent capable of deciding which tool to use based on the user's prompt:
- `search_handbook`: Searches the ChromaDB vector store for policy-related questions.
- `buy_item`: Processes mock purchases from the company store and checks inventory levels.
- `No Tool`: For casual greetings or out-of-scope questions, the model bypasses tools and responds directly or refuses the request.

## Project Structure
- `streamlit_app.py`: The frontend chat interface built with Streamlit.
- `grounded_assistant.py`: The core backend engine handling the RAG pipeline, tool calling, and the LLM client logic.
- `masar_utils.py`: Utility module for tracking token usage, estimating costs, and initializing API clients.
- `data/company_handbook.md`: The source-of-truth document containing Acme Cloud's policies and inventory.
- `REPORT.md` & `eval_results.csv`: Documentation of the multi-model evaluation process and test case results.
- `requirements.txt`: Python dependencies required to run the project.

## Local Setup and Execution

1. **Install Dependencies:**
   Install the required Python packages using pip:
   ```bash
   pip install -r requirements.txt
   ```

2. **Environment Configuration:**
   Create a `.env` file in the root directory and define the required API keys:
   ```env
   OPENAI_API_KEY=your_groq_api_key
   HF_TOKEN=your_huggingface_token
   HF_BASE_URL=https://router.huggingface.co/v1
   HF_MODEL=openai/gpt-oss-20b:groq
   ```

3. **Run the Application:**
   Launch the Streamlit interface by running:
   ```bash
   streamlit run streamlit_app.py
   ```
